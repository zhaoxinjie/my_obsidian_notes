# 工具（tool）
## 定义
在 agent 系统里，`工具（tool）` 不只是一个 API 包装层，也不只是“模型可以调用的函数”。

更准确地说，工具是连接确定性系统与非确定性 agent 的契约层。它决定 agent 能做什么、如何做、做到什么程度，以及它在失败时会得到什么反馈。

## 为什么重要
agent 的有效性，很大程度上取决于它拿到的工具是否适合它使用。

传统软件接口主要服务于人类开发者或其他确定性程序；而 agent 工具面对的是会误解、会犹豫、会选错路径的非确定性调用者。因此，工具设计不能照搬普通 API 设计，而要围绕 agent 的决策方式、上下文限制和常见误用来设计。

## 我们目前对它的几个关键判断
### 1. 工具是 agent 与世界之间的行动契约
- 普通函数调用强调输入输出一致性
- agent 工具除了输入输出，还要考虑模型是否理解它、何时会调用它、调用后是否会被结果带偏
- 所以工具不仅暴露能力，也在塑造策略空间

### 2. 工具工程的目标不是“暴露全部能力”，而是“帮助 agent 更容易走出正确策略”
- 工具过多、过碎、边界重叠，会让 agent 更容易混乱
- 好工具通常围绕高价值工作流设计，而不是机械映射底层接口
- 因此，“该实现哪些工具”与“哪些能力不该直接暴露”同样重要

### 3. 工具返回值本身就是上下文工程
- agent 读取工具结果时，消耗的是上下文预算
- 返回太多低信号字段，会稀释真正有用的信息
- 所以返回值设计、字段选择、详细程度控制，本质上都属于上下文工程

### 4. 工具描述、参数名和错误信息，本质上也是 prompt
- agent 对工具的理解，很大程度来自名字、说明、参数结构和报错反馈
- 模糊参数名、低质量说明、无意义报错，都会直接拉低工具使用质量
- 因此，工具规范本身也是 agent 行为引导的一部分

### 5. 工具质量必须通过评测来迭代
- 很多看起来“设计合理”的工具，实际并不适合 agent 使用
- 是否好用不能靠直觉判断，必须看真实任务下的调用表现
- evaluation 驱动是工具工程的重要基础

## 核心原则
### 1. 少而精
- 不要默认工具越多越好
- 优先实现少量高价值、边界清晰的工具
- 避免多个工具做几乎相同的事

### 2. 面向工作流，而不是面向底层接口
- 工具应尽量贴近任务目标
- 与其暴露一串底层操作，不如提供能直接支持常见任务路径的高层工具
- 这样能减少 agent 的中间推理负担

### 3. 返回高信号信息
- 返回值优先保留会影响下一步决策的内容
- 尽量少暴露对 agent 没帮助的低层技术字段
- 必要时支持不同详细级别，而不是始终返回冗长结果

### 4. 命名清晰、边界明确
- 工具名应直接反映功能
- 命名空间能帮助 agent 在大量工具中快速定位
- 模糊或重叠的名字会增加选错工具的概率

### 5. 报错要能指导下一步行为
- 错误提示不应只是抛出技术细节
- 更好的做法是告诉 agent：哪里错了、应该怎样改、下一步该怎么试
- 对 agent 来说，错误信息是行为引导，不只是日志

### 6. 用评测驱动优化
- 通过真实任务评测观察 agent 如何选工具、如何失败、哪里浪费 token
- 根据结果迭代工具描述、参数结构、返回值和错误信息
- 工具工程不是一次性设计，而是持续改进过程

## 常见优化方向
### 1. 工具选择优化
- 砍掉低价值或高度重叠的工具
- 合并频繁串联调用的多步接口

### 2. 命名与说明优化
- 调整工具名、参数名和描述，使其更符合 agent 理解方式
- 用更明确的语义替代模糊缩写或内部术语

### 3. 返回值优化
- 增加筛选、分页、截断、范围选择
- 用更适合 agent 的格式返回结果
- 在详细与简洁之间提供可控切换

### 4. 错误反馈优化
- 把报错从“技术失败”改成“可行动建议”
- 让 agent 知道如何修正输入或缩小查询范围

## 与其他概念的关系
- 它是 [[wiki/concepts/构建高效智能体|构建高效智能体（Building Effective Agents）]] 的执行层。那一页强调工具重要，这一页回答“工具到底该怎么设计”。
- 它和 [[wiki/concepts/上下文工程|上下文工程（context engineering）]] 强相关，因为工具返回值、工具描述和命名本身就在塑造上下文。
- 它和 [[wiki/concepts/harness工程|harness工程]] 的关系是：前者关注行动接口设计，后者关注长时运行、交接与纠偏。
- 它也是 [[wiki/concepts/LLM-based Agents|大语言模型智能体（LLM-based Agents）]] 的基础组成之一，因为很多 agent 的真实能力边界由工具而不是模型本身决定。

## 对知识库场景的启发
- 对知识库 agent 来说，工具不应只是“读取所有笔记”
- 更好的工具往往是：按主题搜索、读取局部页面、汇总某类来源、提取最近变更
- 如果工具直接返回整个 vault，大概率会让 agent 被噪声淹没
- 因此，好的知识库工具应该帮助 agent 快速定位、局部展开、逐步收敛

## 张力与开放问题
- 工具越高层越方便，但也可能牺牲灵活性，最佳平衡点在哪里？
- 什么时候应该合并工具，什么时候应该保留细粒度能力？
- tool description 的优化收益，和底层实现优化相比，哪个更大？
- 在非 MCP 场景里，工具工程是否会有不同最佳实践？
