# 上下文工程（context engineering）
## 定义
`上下文工程` 不是单纯优化提示词，而是系统性设计“每一轮推理时，模型应该看到什么信息、不该看到什么信息，以及这些信息如何被持续整理和更新”。

它处理的是上下文状态，而不只是提示词文本。

## 为什么重要
很多智能体失败，不是因为模型不会推理，而是因为上下文脏了、乱了、重了、旧了。

上下文是有限资源，不是越多越好。随着 token 增长，模型会逐渐失焦、混淆，出现上下文腐化（context rot）一类的问题。因此，关键不在于“塞进更多信息”，而在于“给模型最小但高信号的信息集合”。

## 我们目前对它的几个关键判断
### 1. 它是提示词工程的上位概念
- 提示词工程关注怎么写 prompt
- 上下文工程关注怎么组织整套输入状态
- 在多轮 agent 系统里，系统提示、工具定义、消息历史、外部记忆、检索结果、运行痕迹，都会共同构成上下文

### 2. 上下文更像工作记忆，而不是仓库
- 上下文窗口不是越大越好
- 它更像一份有限的注意力预算
- 每加入一段信息，都会挤占模型处理其他关键信息的能力

### 3. 高质量 agent 的关键不是“全塞进去”，而是“按需取用”
- 预先加载所有可能相关的信息，常常会让模型被噪声淹没
- 更好的方法是保留轻量引用，在需要时即时拉取
- 因此，文件路径、URL、查询语句、索引、目录结构等元信息，本身就是上下文工程的一部分

### 4. 长任务下的上下文管理需要多种策略协同
- `压缩（compaction）` 解决上下文快满时的连续性问题
- `笔记与外部记忆` 解决长期状态持久化问题
- `子代理架构` 解决复杂探索任务中的上下文隔离与并行处理问题

### 5. 它是很多 agent 工程问题的底层解释
- 有时系统看起来像规划失败，其实是上下文质量失控
- 有时系统看起来像工具使用差，其实是工具返回信息进入上下文的方式不对
- 有时系统看起来像长任务不稳定，本质上是上下文工程没有跟上任务跨度

## 核心原则
### 1. 追求最小但高信号的上下文
- 不要默认信息越多越好
- 只保留当前决策真正需要的信息
- 用结构化摘要代替冗长原始内容

### 2. 优先即时获取，而不是预加载一切
- 让 agent 按需探索和读取
- 通过工具在运行时拉取局部信息
- 这通常比一开始塞进海量资料更稳

### 3. 让元信息帮助模型理解信息空间
- 文件名、目录层级、时间戳、标签、索引都很重要
- 好的信息组织结构会显著降低 agent 的探索成本

### 4. 把上下文整理做成循环过程
- 每一轮都要重新考虑：什么该保留，什么该丢弃
- 上下文工程是持续整理，不是一次性准备

### 5. 把上下文之外的存储当成系统的一部分
- 外部记忆、笔记文件、数据库、索引、进度记录都不只是附件
- 它们是 agent 认知系统的延伸

## 常见手段
### 1. 压缩（compaction）
- 当上下文接近上限时，提炼保真摘要并开启新轮次
- 核心难点在于保留什么、丢弃什么

### 2. 结构化笔记
- 用稳定格式记录关键状态、决定、未完成事项
- 让长期任务不必依赖完整历史消息

### 3. 即时上下文（just-in-time context）
- 只先给轻量入口，不把所有正文直接放进去
- 让 agent 在运行中自行读取需要的信息

### 4. 子代理架构
- 把大任务拆成多个局部探索任务
- 子代理消耗自己的上下文，最后只返回浓缩结果给主代理

### 5. 混合策略
- 一部分关键资料预加载
- 更细信息按需获取
- 这通常比纯预检索或纯即时探索更实用

## 检索前的上下文化（contextualization before retrieval）
很多检索失败，并不是因为系统完全没找到相关内容，而是因为找到的 chunk 已经失去了原始文档语境（document context）。

普通切块（chunking）会把内容切成可检索的碎片，但也会带来一个代价：
- 片段本身可能保留了字面相关性
- 却丢失了它在整篇文档里的位置、主题和解释框架

这会导致两类常见问题：
- 检索命中了局部相关片段，但模型读完仍然不知道它在说什么
- 看起来相似的 chunk 被排到前面，真正有用的 chunk 反而没有进入最终上下文

一种更强的做法是，在 embedding 和 BM25 建索引之前，先给每个 chunk 补一小段局部语境说明（local contextualization）。

这个方法的关键不是“多加文本”，而是让每个 chunk 在进入检索系统之前，就已经带上：
- 它来自哪篇文档
- 它在文档中大致扮演什么角色
- 它与当前主题或实体有什么关系

这样做的意义在于：
- 改善检索，而不是只在生成阶段补救
- 让进入上下文窗口的内容天然更容易被模型理解
- 把“检索增强”提前到预处理阶段，而不是全部压到运行时

对知识库来说，这个思路很重要，因为它说明：
- 好的目录结构、页面标题、来源页和主题页，本身就是检索质量的一部分
- 一个片段如果能被清楚地放回来源、主题和页面结构中，就更容易被正确取回

从工程角度看，这条路线常常和以下做法组合：
- contextual embeddings
- contextual BM25
- reranking

它们共同服务的目标不是“尽量多取一些段”，而是“取回本来就更容易被理解的段”。

## 与其他概念的关系
- 它是 [[wiki/concepts/构建高效智能体|构建高效智能体（Building Effective Agents）]] 的底层支撑之一。有效 agent 的前提之一，是有效的信息进入机制。
- 它是 [[wiki/concepts/harness工程|harness工程]] 的上游基础。harness工程更关注跨轮次交接、纠偏和评估；上下文工程更关注每一轮的工作记忆如何组织。
- 它和更广义的智能体系统（agent systems）的关系是：后者描述系统形态，前者描述系统的认知资源管理方式。
- 它也与检索增强系统相关：像 contextual retrieval 这样的方案，本质上是在优化“什么样的内容更适合被取回并进入上下文”。

## 对知识库场景的启发
- 不要把整个资料库一次性塞给模型
- 更适合保留目录、索引、来源页和主题页，让模型按需展开
- `wiki/index.md`、`wiki/log.md`、来源页和概念页之间的结构，本身就可以看作一种上下文工程
- 你的知识库如果结构清晰，agent 不需要“记住所有内容”，只需要“知道去哪里找”

## 张力与开放问题
- 即时获取虽然更省上下文，但会增加运行时延，如何权衡？
- 什么信息适合压缩，什么信息必须保留原文？
- 多代理架构带来的上下文隔离优势，是否总能覆盖协调成本？
- 在知识库、研究、代码、客服等不同任务中，上下文工程的最佳形态是否不同？
